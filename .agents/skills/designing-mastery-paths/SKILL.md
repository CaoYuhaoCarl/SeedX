# designing-mastery-paths

本 Skill 是 Question-to-Mastery Harness 的生成标准。Planner 和 Builder 必须以它为权威依据。

---

## 一、设计哲学

目标不是写一篇“关于某主题的介绍文章”，而是把一个问题、一个新领域、或一个快速掌握需求，转化为一套 **可执行的掌握路径**。

优秀输出必须帮助学习者：

1. 界定自己到底要弄懂什么。
2. 快速建立领域主干地图。
3. 知道先学什么、后学什么、暂时不学什么。
4. 通过练习验证自己是否掌握。
5. 把知识迁移到输入中明确的目标、场景、项目、工作、研究、创作、决策或讲解任务中。

禁止只做百科式总结、泛泛建议、资料堆砌。

### 个性化原则

- 默认面向通用学习者。
- 只有原始输入明确提供的学习者背景、应用场景、目标受众和约束，才能进入产物。
- 不得从用户档案、历史记忆、项目作者身份或示例题目中自动推断默认场景。
- 如果输入没有提供背景，写“未指定；按通用学习者处理”。

## 二、支持的输入类型

### 1. 明确问题

例如：我想理解 AI Agent memory 为什么重要。

处理重点：

- 澄清问题边界。
- 拆出必须回答的子问题。
- 建立回答这个问题所需的概念地图。

### 2. 新领域入门

例如：我想快速入门 DSPy。

处理重点：

- 识别领域主干。
- 区分入门必学和高级延伸。
- 给出阶段化学习路径。

### 3. 快速框架

例如：我想在 2 小时内建立模型评估的基本框架。

处理重点：

- 严格时间盒。
- 只保留最高杠杆概念。
- 给出短时可完成的学习路线。

### 4. 决策前学习

例如：我要判断是否用某个框架解决一个真实项目问题。

处理重点：

- 明确决策标准。
- 学习路径服务决策，而非追求完整学科覆盖。
- 输出比较维度和最小验证实验。

---

## 三、好问题标准

一个好学习问题应满足：

1. **有边界**：不是“学习 AI”，而是“理解 AI Agent memory 的基本框架”。
2. **有目的**：说明为什么要学，学完要做什么。
3. **可行动**：能转化为学习阶段、练习和 checkpoint。
4. **可验证**：能定义“学会了”的判断标准。
5. **有应用出口**：能连接到输入中明确的使用场景、项目、工作、研究、创作、决策或讲解任务；如果未指定，则保持通用应用出口。

如果原始问题过宽，Planner 必须在 `learning-contract.md` 中收窄并写明边界。

## 四、Learning Contract Schema

`learning-contract.md` 必须包含：

```markdown
# Learning Contract

## 原始输入
{输入文件路径与原始问题摘要}

## 问题类型
明确问题 / 新领域 / 快速框架 / 决策前学习

## 学习者背景与应用场景
{只提取输入中明确写出的背景、目标、场景、受众和约束；若未指定，写“未指定；按通用学习者处理”}

## 最终学习目标
{学完后应能理解、判断或做出的事情}

## 必须回答的问题
1. ...
2. ...
3. ...

## 必须覆盖的核心概念
1. ...
2. ...
3. ...

## 暂不覆盖
1. ...
2. ...

## 固定任务单元
- Task 1: Framing — question-brief.md + domain-map.md
- Task 2: Mastery Path — learning-path.md + exercises.md + checkpoints.md
- Task 3: Application & Transfer — application-plan.md + transfer-plan.md

## 合格标准
1. 问题边界清晰。
2. 领域主干不遗漏。
3. 学习路径有阶段、有顺序、有时间感。
4. 练习具体可做。
5. checkpoint 可验证。
6. 应用与迁移计划贴合输入中明确的目标或场景；若未指定，保持通用且可执行。
7. 不引入输入中未提供的个人、行业、职业或产品背景。
```

## 五、任务单元设计

MVP 固定 3 个任务单元，每个任务单元由主 Agent 单独启动 Builder 和 Evaluator。

### Task 1: Framing

产物：

- `question-brief.md`
- `domain-map.md`

目标：回答“我要学什么”和“这个领域的主干结构是什么”。

必须包含：

- 原始问题重述。
- 学习目标。
- 明确边界和非目标。
- 核心概念树。
- 前置知识。
- 常见误区。

### Task 2: Mastery Path

产物：

- `learning-path.md`
- `exercises.md`
- `checkpoints.md`

目标：回答“我如何一步步掌握”和“如何验证掌握”。

必须包含：

- 阶段化路径。
- 每阶段目标。
- 每阶段不学什么。
- 具体练习任务。
- 可验证 checkpoint。

### Task 3: Application & Transfer

产物：

- `application-plan.md`
- `transfer-plan.md`

目标：回答“这个知识如何变成行动、判断、项目、工作成果或可讲解内容”。

必须包含：

- 输入中明确目标/场景下的应用路径；若未指定，给出通用应用路径。
- 最小实验或最小下一步。
- 可交付物或行动清单。
- 面向目标受众的解释方式；若未指定，默认面向自己、同伴或非专业听众。
- 类比、例子、练习、项目步骤或决策清单转化。

## 六、领域地图设计原则

领域地图不是目录堆砌。必须区分：

1. **主干**：理解该领域不可缺少的概念。
2. **分支**：理解主干后可以扩展的方向。
3. **边缘话题**：本轮不需要深入。
4. **前置知识**：缺少会影响理解的基础。
5. **常见误区**：学习者容易混淆或误判的地方。

表达可以用层级列表、表格、概念关系说明。不要为了完整而塞入低价值术语。

---

## 七、学习路径设计原则

学习路径必须有阶段感：

```text
Stage 1: 建立边界和基本概念
Stage 2: 理解核心机制
Stage 3: 通过练习应用
Stage 4: 迁移到目标场景或通用输出任务
```

每个阶段必须写：

- 目标。
- 建议耗时。
- 学习内容。
- 不学什么。
- 输出或练习。
- 完成标准。

---

## 八、练习设计原则

练习必须具体，避免“多阅读资料”“深入研究”这类空话。

优先使用：

1. 比较：比较 A/B/C 的区别。
2. 解释：用自己的话解释给指定受众；未指定时解释给非专业听众。
3. 设计：设计 schema、流程、实验、项目步骤或决策表。
4. 应用：把概念用于输入中明确的目标场景；未指定时用于通用小项目或真实问题。
5. 反例：识别错误理解或失败案例。
6. 最小实验：当天可完成的小 demo、小表格、小决策或小输出。

---

## 九、Checkpoint 设计原则

Checkpoint 必须能验证掌握，不是情绪化评价。

好的 checkpoint：

- “能区分 context、short-term memory、long-term memory、user profile，并各举一个通用应用例子。”
- “能写出 15 条信息，并分类为应长期保留/临时上下文/不应记忆。”
- “能设计一个最小 schema，并说明更新规则。”

坏 checkpoint：

- “我感觉理解了。”
- “看完三篇文章。”
- “继续深入学习。”

---

## 十、应用与迁移原则

每个学习包都要回答：

1. 这个知识能帮助学习者完成什么行动、判断或输出？
2. 如何用于输入中明确的目标场景？如果未指定，如何用于通用场景？
3. 如何解释给指定受众？如果未指定，如何解释给非专业听众？
4. 能否变成一个练习、实验、项目步骤、决策清单、讲解稿或作品？
5. 最小下一步是什么？

应用计划必须具体，不要写泛泛的“可用于某行业”。
